# Phase 7 Plan 3: Keyboard Actions - HID Mapping and Key Press Summary

**One-liner:** Complete HID key code mapping with single key and combination support, including comprehensive unit tests.

---

## What Was Built

### 1. HID Key Mapping Table (KEY_MAP)

Complete mapping of key names to HID Usage Table 1.5 codes:

- **Letters a-z:** Codes 4-29
- **Numbers 0-9:** Codes 30-39
- **Special keys:** enter (40), esc (41), backspace (42), tab (43), space (44)
- **Arrow keys:** up (38), down (40), left (37), right (39)
- **Modifiers:** alt/lalt (226), ralt (230), ctrl/lctrl (224), rctrl (228), shift/lshift (225), rshift (229)
- **F keys F1-F12:** Codes 58-69

### 2. Key Name Parsing (_parse_key)

Parses key names into HID codes:
- Single keys: `"esc"` -> `[41]`
- Combinations: `"alt+u"` -> `[226, 24]`
- Case-insensitive: `"ALT+U"` works
- Whitespace tolerant: `"alt + u"` works
- Raises `ValueError` for unknown keys

### 3. Key Press Method (press)

Handles both single keys and combinations:
- **Single key:** Uses `km.press(code)` with auto-timing (75-125ms press, 125-175ms release)
- **Combination:** Uses `km.down()` sequence with 10ms delays, 50ms hold, then `km.up()` in reverse

### 4. Modifier Ordering (_order_codes)

Ensures correct chord sequence:
- Separates modifiers from main keys
- Returns modifiers first (e.g., `"u+alt"` sends alt before u)
- Critical for hotkey combinations like Alt+U for guild menu

### 5. Hold/Release Methods (key_down/key_up)

For scenarios requiring manual key timing:
- `key_down("alt")` -> sends `km.down(226)`
- `key_up("alt")` -> sends `km.up(226)`
- Must be paired to avoid stuck keys

### 6. Test Suite

18 tests covering:
- KEY_MAP completeness (7 tests)
- Key parsing logic (4 tests)
- Modifier ordering (3 tests)
- Integration tests (4 tests, require hardware)

All 14 unit tests pass without hardware.
